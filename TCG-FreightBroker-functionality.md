# TCG Freight Broker Logic Workbench — Functionality Overview

> Generated by functionality-extractor. Use this as a requirements reference when rebuilding in a new framework.

## Project Summary
This prototype is a single-page freight brokerage decision workbench used to simulate, evaluate, and manage load-booking decisions with a contract-first strategy. It combines an automated decision engine with manual override workflows, lane obligation tracking, client-specific GP targets, user roles, and full audit logging. The main business goal is to protect margin while still meeting weekly contract lane commitments.

## Detected Stack
- Language: JavaScript, HTML, CSS
- Framework: None (vanilla browser app, no build system)
- Runtime model: In-memory client-side state only (no backend or database)
- Notable dependencies: Google Fonts only

## Features

### Load Generation And Intake
**What it does:** Creates synthetic freight loads that mimic incoming opportunities from DAT or e2open sources.
**Inputs:** Lane catalog (contract and spot), weekly booking status, randomization factors, carrier list, lane-to-client mapping.
**Outputs:** New load objects with lane, carrier cost, customer rate, profit, margin, contract GP, source, and initial status.
**Key behaviors:**
- Prioritizes unmet contract lanes with weighted selection when obligations are behind.
- Produces both contract and spot opportunities.
- Attaches client code automatically when lane is assigned to a client.

### AI Decision Engine
**What it does:** Evaluates each load against business rules and recommends auto-accept, contract-book, or review.
**Inputs:** Load financials, lane type, weekly contract progress, global tuning parameters, client-specific settings.
**Outputs:** Rule audit trail, weighted score, recommendation, GP-block flag, and pass/fail summary.
**Key behaviors:**
- Enforces contract-rate GP floor (global or per-client override).
- Applies different margin thresholds for contract-need, normal contract, and spot lanes.
- Supports contract-obligation overrides for negative profit / low margin (configurable).
- Suppresses auto-accept when client auto-accept is disabled.
- Adds urgency escalation when contract loads are behind near week end.

### Auto-Booking Loop
**What it does:** Continuously generates and auto-processes loads while auto mode is enabled.
**Inputs:** Timer tick, AI evaluation output, current booking state.
**Outputs:** Accepted/rejected/pending loads, board updates, audit events, optional e2open open action.
**Key behaviors:**
- Auto-accepts recommendations of AUTO-ACCEPT and CONTRACT-BOOK.
- Auto-rejects clearly bad review cases (negative profit or negative contract GP), otherwise leaves in pending review.
- Maintains running load/event streams for UI dashboards.

### Decisions Queue And Manual Operations
**What it does:** Provides an operational queue to review pending/rejected/accepted loads and execute manual decisions.
**Inputs:** Current loads, selected filters/sorts, AI recs, multi-select actions.
**Outputs:** Manual accept/reject actions, changed load statuses, updated metrics.
**Key behaviors:**
- Supports sorting by time, lane, GP percent, and profit.
- Supports AI-assisted bulk selection (select all AI-accept or AI-reject candidates).
- Includes batch confirmation bar before bulk commit.
- Includes accepted-load day drilldown (daily performance view).

### Confirmation Safety Controls
**What it does:** Prevents accidental decisions with countdown-based confirmations.
**Inputs:** User accept/reject intent.
**Outputs:** Timed confirmation prompts, cancelable action windows.
**Key behaviors:**
- Single-load confirmation countdown (20 seconds).
- Batch-level confirmation workflow for mass actions.

### Contract Tracker Dashboard
**What it does:** Monitors weekly contract fulfillment and operating performance.
**Inputs:** Weekly lane minima, accepted contract counts, current week context.
**Outputs:** Fill percentage, lane progress bars, risk indicators, KPI cards.
**Key behaviors:**
- Highlights critical lanes when remaining days are low and required pace is high.
- Separates weekly-minimum lanes from accept-all lanes.
- Shows gross profit and current accepted/rejected/pending counts.

### Load Board
**What it does:** Displays live list of loads with status, financials, and quick actions.
**Inputs:** All load records and current board filter.
**Outputs:** Tabular operational board and quick accept/reject controls.
**Key behaviors:**
- Shows latest loads first.
- Supports quick one-click actions with safety confirmation layer.
- Opens detailed modal per load.

### Load Detail And Rule Explainability
**What it does:** Provides per-load drilldown with decision explanation.
**Inputs:** Selected load.
**Outputs:** Full financial breakdown, rule-by-rule outcome, final recommendation context.
**Key behaviors:**
- Exposes GP block rationale clearly.
- Displays client-specific GP target used in decision.
- Allows decision actions directly from detail view.

### Lane Drilldown Analytics
**What it does:** Provides lane-level diagnostics across activity and profitability.
**Inputs:** Selected lane, lane profile, session load history.
**Outputs:** KPI cards, carrier performance summary, GP distribution, recent loads.
**Key behaviors:**
- Compares lane outcomes against GP floor.
- Displays session-level accepted/rejected/pending/blocked totals.
- Supports quick jump to lane testing or board workflow.

### Manual Load Tester
**What it does:** Lets operators test hypothetical load scenarios against the AI logic.
**Inputs:** User-entered lane, rates, carrier, and spot/contract values.
**Outputs:** Live recommendation, score, full rule trace, financial indicator cards.
**Key behaviors:**
- Lane selector auto-populates baseline rates.
- Random test mode generates realistic sample instantly.
- Preserves last test output until cleared.

### Parameter Tuning Workspace
**What it does:** Lets admins tune decision thresholds and override behavior.
**Inputs:** Slider/toggle controls for GP floor, margin floors, DAT tolerance, urgency thresholds, and override flags.
**Outputs:** Updated runtime decision behavior.
**Key behaviors:**
- Global GP floor is overridden per client when configured.
- Includes reset-to-default parameter set.

### Batch Simulation
**What it does:** Runs large synthetic batches to estimate policy outcomes before deployment.
**Inputs:** Batch size, current tuning parameters, lane/client setup.
**Outputs:** Distribution of outcomes, GP totals, by-client and by-lane result tables.
**Key behaviors:**
- Separates accepted vs review vs rejected populations.
- Quantifies GP-block frequency.
- Reports accepted-dollar and rejected-dollar impact.

### DAT Rates Reference View
**What it does:** Shows all lanes with contract/spot/historical indicators in one table.
**Inputs:** Contract and spot lane catalogs, client assignments.
**Outputs:** Readable lane reference matrix for planning.
**Key behaviors:**
- Includes lane type, weekly minimum, contract/spot rates, average revenue/profit, YTD totals.
- Supports direct lane drilldown.

### Client Management
**What it does:** Maintains customer profiles and decision constraints.
**Inputs:** Client code/name/color, GP target, margin floor, auto-accept flag, lane assignments, notes.
**Outputs:** Updated client-to-lane policy mapping and display context.
**Key behaviors:**
- One lane belongs to at most one client assignment at a time.
- Unassigned lanes fall back to global GP policy.
- Shows client-level runtime performance snapshots.

### Integrations Configuration
**What it does:** Captures integration settings for DAT API, email ingestion mode, and e2open acceptance links.
**Inputs:** API credentials/endpoints, provider mode, polling settings, e2open URL.
**Outputs:** Integration status state (connected/disconnected), saved local settings.
**Key behaviors:**
- Supports DAT sandbox/production setting.
- Supports email modes: e2open API, IMAP, manual.
- Optional behavior to open e2open in browser on load acceptance.

### Config Export And Import
**What it does:** Backs up and restores operational settings.
**Inputs:** Current in-memory configuration state; imported JSON file.
**Outputs:** Downloadable JSON export or applied imported state.
**Key behaviors:**
- Exports clients, integrations, parameters, users, and e2open URL.
- Import merges state into active runtime.

### User Authentication And Role-Based Access Model
**What it does:** Provides lightweight PIN-based login and role categorization.
**Inputs:** Selected user and PIN.
**Outputs:** Active user context for attribution and permission intent.
**Key behaviors:**
- Roles defined: admin, broker, viewer.
- Captures login/logout timestamps.
- Role permissions are represented in UI documentation and helper checks.

### User Administration
**What it does:** Adds, edits, activates/deactivates, and deletes users.
**Inputs:** Username, display name, email, role, PIN, active toggle.
**Outputs:** Updated user roster and audit events.
**Key behaviors:**
- Tracks creation and last login fields.
- Generates user IDs incrementally.

### Audit Logging And Compliance View
**What it does:** Records operational and admin events with searchable history.
**Inputs:** System and user actions (accept/reject, login, config changes, exports/imports, e2open opens).
**Outputs:** Structured audit entries with timestamp, actor, action, load ID, GP dollars and percent.
**Key behaviors:**
- Filter and search support.
- Drilldown by user for manual decisions.
- CSV export capability.

### Step-Through Decision Mode
**What it does:** Training/explainability workflow that processes one load at a time with full rule visibility.
**Inputs:** Generated load and user accept/reject/skip choice.
**Outputs:** Explicit decision history and AI-vs-user agreement metrics.
**Key behaviors:**
- Shows each rule outcome with weight contribution.
- Tracks accepted/rejected/pending user action against AI recommendation.
- Produces agreement percentage and override counts.

### Header And Operational Controls
**What it does:** Exposes current fiscal week context, fill score, and auto mode toggle.
**Inputs:** Date/time, weekly booking status, auto flag.
**Outputs:** Global operational status display.
**Key behaviors:**
- Computes current week window and days remaining.
- Displays dynamic fill percentage coloring.

## Data And State
Primary runtime entities are stored in browser memory only.

- Lane profile: contract/spot characteristics including rates, weekly targets, historical metrics, and notes.
- Load: unique shipment decision object with financials, AI rec, source, status, timestamps, and optional client assignment.
- Weekly booking state: accepted contract count per lane for obligation tracking.
- Client: policy envelope for assigned lanes (GP targets, auto-accept, notes).
- User: login identity with role and PIN.
- Integration settings: DAT/email/e2open connectivity configuration.
- Parameters: tunable rule thresholds and override flags.
- Audit entry: immutable event trail with actor, action, and GP impact.
- Step history: training-mode history of AI and user decisions.

## External Integrations
- DAT Rateview API: intended for spot/contract rate comparison and sync.
- e2open: optional browser URL open on accepted loads; also represented as email/API provider mode.
- IMAP email: optional mailbox connectivity for appointment/order ingestion workflow.

## Configuration And Environment
- Global decision thresholds: GP floor, margin floors, DAT tolerance, urgency criteria.
- Contract override toggles: allow below-margin or negative-profit acceptance when obligations demand.
- Client-level policy overrides: GP target and auto-accept control by account.
- Integration mode and credentials: DAT API key/secret, email provider credentials, e2open URL.
- Security model: local PIN-based authentication and role assignment.

## Notes For Rewrite
- Current implementation is a frontend-only prototype with all state in memory; persistence, multi-user concurrency, and true permission enforcement are not yet backend-enforced.
- Integration tests are simulated with status toggles and alerts; production rewrite should implement real API calls, retry logic, failure handling, and credential security.
- The decision engine is the core business asset and should be extracted into a testable domain service with deterministic unit tests.
- Batch simulation, tester, and step-through workflows are valuable for explainability and should be retained as first-class tooling.
- Audit logging is strong conceptually; rewrite should preserve event richness and add tamper-resistant storage.
- Backtest section appears started but not fully implemented in the provided file snapshot; historical replay requirements should be clarified during rebuild scoping.
