# Functionality Extractor

 

Reads an existing codebase from a local folder path and produces a structured Markdown document describing what the project does — its features, data flows, and key behaviors — at a high level (what, not how). The output is designed to serve as a requirements document for rebuilding the project in a new framework or language.

 

---

 

## Step 1: Ingest the Project

 

The user will provide a local folder path. Start by getting oriented:

 

```bash
# Get a high-level view of the structure
find <path> -type f | grep -v -E "(node_modules|\.git|bin|obj|dist|__pycache__|\.vs)" | head -100
```

 

If the tree is large, focus on:
- Top-level files (README, config files, entry points)
- Source directories (src/, app/, lib/, Controllers/, Services/, etc.)
- Key config files (package.json, .csproj, pyproject.toml, appsettings.json, etc.)

 

---

 

## Step 2: Auto-Detect Framework and Language

 

Look at the file structure and config files to identify:

 

| Clue | Likely Stack |
|---|---|
| `package.json` with react/vue/next | JS frontend framework |
| `.csproj` / `Program.cs` / `Startup.cs` | .NET / C# |
| `pyproject.toml` / `requirements.txt` / `*.py` | Python |
| `Worker.cs` / `BackgroundService` | .NET Worker Service |
| `Controllers/` folder | MVC backend (ASP.NET, etc.) |
| `appsettings.json` | .NET |
| `Hangfire` references | .NET background jobs |

 

State your detection at the top of the output doc. If ambiguous, note that too and proceed with best guess.

 

---

 

## Step 3: Read Key Files

 

Read strategically — you don't need every file, just enough to understand what the system does.

 

**Priority reading order:**
1. Entry point (Program.cs, index.js, main.py, App.tsx, etc.)
2. Config files (appsettings.json, package.json, .env.example)
3. Route/controller/page definitions — these reveal the feature surface
4. Service/business logic classes — these reveal what each feature actually does
5. Database schema or models — reveals data shape
6. Any README or existing docs

 

**Skip:** migrations, auto-generated files, test files (unless they clarify behavior), vendor/library code.

 

---

 

## Step 4: Produce the Functionality Document

 

Write a structured Markdown file with this shape:

 

```markdown
# [Project Name] — Functionality Overview

 

> Generated by functionality-extractor. Use this as a requirements reference when rebuilding in a new framework.

 

## Project Summary
One paragraph: what this project is, who/what it serves, and its primary purpose.

 

## Detected Stack
- Language: ...
- Framework: ...
- Notable dependencies: ...

 

## Features

 

### [Feature or Module Name]
**What it does:** A plain-English description of this feature's purpose and behavior.
**Inputs:** What triggers it or what data it receives (user action, scheduled job, API call, file, etc.)
**Outputs:** What it produces or causes (response, file, DB write, downstream call, etc.)
**Key behaviors:** Any notable rules, conditions, or flows worth preserving in the rewrite.

 

(Repeat for each meaningful feature/module)

 

## Data & State
What data does the system store or manage? Describe the main entities/models in plain terms — not schema, just what they represent and their relationships.

 

## External Integrations
List any third-party services, APIs, or systems the project talks to, and what it uses them for.

 

## Configuration & Environment
Key settings the system depends on (auth, connection strings, feature flags, etc.) — without actual values.

 

## Notes for Rewrite
Anything worth flagging: tricky business logic, things that were clearly workarounds, areas that seem fragile or over-engineered.
```

 

---

 

## Step 5: Save and Deliver

 

Save the document to the project folder (or alongside it):

 

```
<original-path>/../<project-name>-functionality.md
```

 

Then present it to the user with `present_files`.

 

---

 

## Principles

 

- **What, not how.** Describe behavior and purpose, not implementation details. The reader should be able to rebuild from this doc without the original code.
- **Be concrete.** Vague descriptions like "handles data processing" are useless. Name what data, what processing, what outcome.
- **One feature per section.** If a module does three things, write three sections.
- **Flag ambiguity.** If you can't tell what something does from reading it, say so — don't guess silently.
- **Don't over-read.** You don't need to read every file. Get enough signal to describe the system accurately, then stop.
